# RFC — Extension trust, provenance and permissions

- **Date:** 2026-07-12
- **Issue:** [#12](https://github.com/matbrgz/blackfin/issues/12) — *RFC: Extension trust, provenance and permissions*
- **Status:** **RATIFIED** by the maintainer. **D1 = disclosure over containment**: Blackfin shows what an extension adds, where it came from, and what it does not know, and requires explicit per-install review — but **never asserts an extension is safe**. A Blackfin sandbox and binary trust are rejected; real containment is delegated to the agent's own permission model; reputation/allowlist is policy (#53), never a safety claim. The rest of the register (D2–D8) follows from D1. #50/#51/#53/#38/#65 implement against this.
- **Depends on:**
  - [#10](https://github.com/matbrgz/blackfin/issues/10) — *taxonomy* (**RATIFIED**, `docs/superpowers/rfcs/2026-07-12-taxonomy.md`): the `Extension` / `Capability` / `requires` vocabulary this document hangs trust on.
  - [#11](https://github.com/matbrgz/blackfin/issues/11) — *extension domain model* (**RATIFIED**, `docs/superpowers/rfcs/2026-07-12-extension-model.md`): `source`, `IExtensionRecord`, `contentHashAtInstall`, `IMcpServer.envKeys`, the *never-write-silently* boundary.
  - [#13](https://github.com/matbrgz/blackfin/issues/13) — *marketplace architecture* (**RATIFIED**, `docs/superpowers/rfcs/2026-07-12-marketplace-arch.md`): **federation** — a static Git index that points at origins; **Blackfin hosts nothing, curates nothing, signs nothing.**

  Where this document and the three ratified RFCs touch, **the ratified RFCs win.**
- **Blocks:** #50 (install permission review), #51 (integrity/signatures), #53 (org policy/allowlists), #38/#39 (install from Git/URL), #65 (CLI).
- **Scope of this document:** the trust *posture*; what an extension *declares it can do* (reusing #11, not redefining it); the shape of the install permission review (#50); what provenance is recorded and what is honestly unknowable; and the hard invariants that keep Blackfin an observer. **No production code.** The integrity/signature mechanics are #51; the registry is #13; org policy is #53; UI screens are #50/#49.

---

## 1. Problem

An Extension **is not trusted code by default.** A skill is a Markdown file that instructs
a model to run commands. A hook is literally a script that runs in the user's shell. An MCP
server is a process with access to the user's filesystem, network and credentials.
Installing any of the three from the internet is running a stranger's code on your machine,
with your permissions.

Blackfin is getting an "Install" button (M2, M3). Before it does, it must answer three
questions that have no answer anywhere in the code today:

1. **What is a permission**, when the object is a Markdown that persuades a model, not a
   binary that calls a syscall?
2. **How is provenance established** — where did this come from, who wrote it, and is it the
   same bit they published?
3. **What can Blackfin actually *prevent*** — and what can it only *warn* about?

And #13's ratification sharpens the question past the point where the comfortable answers
survive. #13 ratified **federation**: the registry is a static Git index that *points at
origins*; **Blackfin hosts no bytes, curates no submissions, and operates no signing key.**
That is a deliberate, ratified choice — and it removes the one source of trust that other
marketplaces lean on. Trust here **cannot** come from *"Blackfin vouched for it,"* because
Blackfin, by ratified design, vouches for nothing. There is no curation service to bless a
package, no signing authority to attest one. So this RFC has to answer a harder question
than most trust documents: **where does trust come from when the platform explicitly refuses
to be its source?**

## 2. Why it matters

The dangerous moment is not execution — it is the **moment of install**, the only instant
Blackfin is in the path. Once the file is on disk, Blackfin is a spectator: another process
(the agent) reads it and acts on it, on its own lifecycle, with its own permission model and
the user's shell. So every gram of security value this product can deliver has to be spent
right there: in *what the user knows before saying yes*, and in *how hard it is to say yes
by accident*.

The real failure mode is not exotic. It is this: a user sees a useful skill in a
marketplace, clicks Install, and the skill contains a line instructing the agent to read
`~/.aws/credentials` and POST it to an endpoint. No antivirus sees anything. No sandbox
would fire — the agent is *allowed* to read files, and it is the agent that reads. The only
defence that ever existed was the screen the user saw before clicking.

If that screen lies — if it says "sandboxed" or "verified" without a basis — Blackfin has
**increased** the risk: it has handed the user psychological permission to install things
they would, without the badge, have read first. And #13 makes the temptation concrete: a
"from the official marketplace" label reads as a blessing. It is not one. An item from the
federated index is **no safer** than any Git repo; it is only *easier to find*.

## 3. Current state (grounding)

**There is nothing about trust, permission or provenance in the code.** `IContextFile`
(`app/src/models/workspace-inventory.ts:95-114`) has no `source`, no `trust`, no
`permissions`. What exists — and determines what is honest to promise — is:

- **Blackfin does not execute extensions, and has no mechanism to.** Nothing in
  `app/src/lib/workspace/` (`scan.ts`, `scan-global.ts`, `catalog.ts`, `parse.ts`,
  `cleanup.ts`) runs anything. The scanners read and classify; the classification is pure,
  no I/O. This is a virtue, and this RFC treats it as an invariant, not a gap.
- **The renderer runs with full Node privileges.** `app/src/main-process/app-window.ts:74-81`:
  ```ts
  webPreferences: {
    disableBlinkFeatures: 'Auxclick',
    nodeIntegration: true,
    spellcheck: true,
    contextIsolation: false,
  },
  ```
  `nodeIntegration: true` and `contextIsolation: false`. **The Electron sandbox is already
  off in this app** — inherited from GitHub Desktop, ten years ago — and turning it on is a
  months-long project of its own that rewrites the whole main/renderer boundary. Any proposal
  to "sandbox plugins with the Electron sandbox" dies here, in two lines of config.
- **The only "sandbox" in the app renders Markdown.** `app/src/ui/lib/sandboxed-markdown.tsx:395`
  — `sandbox="allow-same-origin"` on an `<iframe>`. It isolates *HTML rendering*, not
  *extension execution*. It is not reusable for any of this, and naming "sandbox" twice in
  one product with two meanings is a communication trap.
- **The subprocess precedent is the opposite of containment.**
  `app/src/lib/stores/copilot-store.ts:766` re-executes Electron itself with
  `ELECTRON_RUN_AS_NODE: '1'` to run the Copilot CLI: a Node process with **all** the user's
  permissions. That is how the app already talks to an agent, and it is the proof that the
  execution model here is full trust, not containment.
- **The safe-destruction doctrine already exists, and is the best material for this RFC.**
  `app/src/lib/workspace/cleanup.ts:55-123` (`checkDeletable`): lexical path containment,
  **refusal to follow symlinks** (`:84`), reclassification from scratch at the instant of
  deletion (`:102-120`), trash instead of `rm -rf` (`:142-146`), and failure as a *result*
  not an exception (`CleanupOutcome`, `:22-33`). The rigour that function applies to deleting
  a `node_modules` is the floor for the rigour installing an extension demands.
- **The secret boundary already exists as pure functions.**
  `app/src/lib/workspace/mcp/secrets.ts` (#45/#86): `classifyEnvVar` computes a *presence*
  from the *shape* a config declared — never the value; the input type `DeclaredEnvValue` has
  no room for a value, and `environmentHasName` asks whether a **key** exists (`!== undefined`)
  without ever reading it. `sanitizeMcpUrl` strips userinfo, query and credential-shaped path
  segments from a URL. **Nothing in that file reads a secret, and nothing in it can return
  one.** This RFC makes that boundary a rule of the trust model, not an implementation detail.

## 4. The trust model — where trust comes from when the platform refuses to be its source

This is the central decision (D1). Because #13 ratified that Blackfin curates nothing, signs
nothing and hosts nothing, the trust posture cannot be inherited from the platform — it has
to be *built out of what Blackfin can honestly assert*. Below are the candidate sources of
trust, surveyed and judged.

### 4.1 The temptation this section exists to kill — a Blackfin-imposed sandbox

The comfortable answer to "what can Blackfin prevent?" is *"we'll sandbox it."* **Blackfin
cannot sandbox anything, and this is not a temporary limitation — it is a structural property
of the product:**

> **Blackfin never executes an extension. The agent does.**

Blackfin writes a `SKILL.md` into `~/.claude/skills/`. After that, the agent — another
process, with its own lifecycle, its own permission model and the user's shell — reads that
file and does what it says. No mechanism inside an Electron app can intercept that. The
demonstration, so this proposal is not reproposed in six months:

1. **Blackfin does not execute the extension.** It writes a file. Whoever reads and acts on
   it is the agent, in another process, started by the user, outside our lifecycle. There is
   no interception point. Sandboxing Blackfin's own process contains nothing — Blackfin's
   process does nothing dangerous.
2. **The dangerous object is not code; it is *persuasive text*.** A malicious skill does not
   call a syscall — it **convinces a model** to call one. No process sandbox intercepts
   persuasion. The effective boundary is the *agent's* permission model (Claude Code's
   `allowed-tools`, for example), which is the agent's, not ours.
3. **The Electron sandbox is already off in this app** (`app-window.ts:74-81`:
   `nodeIntegration: true`, `contextIsolation: false`), inherited from ten years of GitHub
   Desktop. Turning it on is a standalone months-long project — and even finished, it changes
   nothing in (1) and (2).
4. **Real OS containment** (Seatbelt, AppArmor, containers) **would have to wrap the agent's
   process**, which Blackfin does not start, does not manage and must not manage — doing so is
   becoming an agent host, which is the thing the product decided not to be
   (`docs/BRIEFING.md` §3).

**Verdict: a Blackfin-imposed sandbox is rejected, with this demonstration.** Not because it
is hard — because it is *incoherent*. A sandbox offered here would be a **security lie**: a
seal that grants confidence without delivering containment. That is worse than no seal.

### 4.2 Binary trust — installed = trusted

No model. You installed it, you trust it. Like `npm install`.

- **For:** zero friction, zero code, honest by omission (promises nothing).
- **Against:** it is exactly the model that produced every npm/PyPI/VS Code supply-chain
  incident of the last decade. And the target here is worse: a developer's machine with
  production credentials on disk and an agent authorised to run shell. It also throws away
  Blackfin's one real asset — it is the only tool that *sees* the user's whole context surface
  and could show it before the click.
- **Verdict:** rejected. Not because it is insecure — because it wastes the only thing Blackfin
  can offer.

### 4.3 The three real candidate sources of trust

With sandbox and binary-trust off the table, trust can come from one (or a combination) of:

#### (A) Trust-on-install, via an explicit permission review

The user is shown exactly what the artifact is and does, and approves it once, per install,
at the moment of install. Trust is an act the user performs, informed by disclosure —
Blackfin asserts nothing beyond what it read.

- **For:** it is the only moment Blackfin is in the path (§2). It puts the whole security
  budget where it can be spent. It requires Blackfin to *assert* nothing about safety — only
  to *show* accurately.
- **Against:** it depends on the user reading. A determined clicker installs anyway. This is
  the real ceiling of the product, and the RFC states it out loud rather than hiding it behind
  a badge.

#### (B) Provenance-based — show where it came from and let the user decide

Blackfin records and shows, for each item: where it came from (origin URL, commit/tag), who
*claims* to have authored it (never verified), whether the source is pinned by an exact
commit, and — critically — what Blackfin *does not know*. The user decides on the strength of
facts, each carrying its own certainty.

- **For:** it is #13's federation posture realised. "Came from `github.com/foo/bar@abc123`" is
  a **stronger** fact than "came from the marketplace," which is a fact about an intermediary.
  It plays to the one thing only Blackfin can do — see the whole surface and say exactly where
  each piece is from.
- **Against:** provenance is not safety. An exact commit from a named author can still be
  malicious. Provenance narrows *who to blame*, not *whether to trust*.

#### (C) A reputation / allowlist layer

Trust derived from stars, download counts, an allowlist, or an organisation's policy — a
signal that *others* (or the org) vouch, since Blackfin will not.

- **For:** it is the only source that scales past the individual user's judgement, and in an
  org it is the right authority (an admin, not the app).
- **Against:** Blackfin measures **no downloads** (#13 §10: no telemetry, nobody to leak to);
  stars are the *origin repo's* stars, a weak and gameable proxy. And an allowlist is a *policy*
  object, not a *trust* object — it belongs to whoever sets policy. **This is #53, not this RFC.**
  Reputation here is a lead, never a verdict.

### 4.4 Recommendation — disclosure over containment; the agent contains, not us

**Adopt (A) + (B) as the trust posture — trust-on-install driven by an explicit permission
review, built on provenance — and treat (C) as a policy layer owned by #53, never a safety
claim.** Sandbox (§4.1) is rejected with demonstration; binary trust (§4.2) is rejected as a
waste of the product's one asset.

Blackfin does not prevent anything at runtime. What it does — better than any other tool — is:
**open the artifact before installing, say exactly what is inside, say exactly where it came
from, say exactly what it does not know, and write to disk only after an explicit yes.** The
founding posture, stated so it reaches the UI verbatim:

> **Blackfin does not execute extensions and cannot contain them. The agent executes them.
> What Blackfin can do — and no other tool does — is open the package before you install it
> and tell you exactly what is inside, where it came from, and what it does not know. It shows
> you what this is; it does not tell you it is safe.**

And there is a fifth, complementary layer — **(D) delegate real containment to the agent.**
The only place containment *actually happens* is the agent's own permission model
(`allowed-tools`, MCP permissions). Blackfin's contribution is to **surface** that model where
it exists — and, more valuable, to say loudly when there is **none**. Blackfin displays it; it
does not reimplement it. This is not an alternative to (A)+(B); it is the honest completion of
it: the one containment that exists is the agent's, and Blackfin points at it instead of faking
its own.

**Normative consequences** (each is a rule downstream issues inherit):

- **No use of the word "sandbox" in the extension UI.** Under any wording. Nor "safe," nor
  "verified" without a direct object (*"verified: the checksum matches what the registry
  published"* is allowed; a bare "Verified" badge is not).
- **`writesTo` is derived by Blackfin, never declared by the author** (§6). A manifest that
  says "I write only here" is a stranger's assertion. The installer *knows* where it writes,
  because it is the one writing.
- **An update that adds a capability requires fresh approval** (§7.3). Non-negotiable.
- **Secrets are a type invariant, not a convention** (§8): the MCP-server type has no field for
  an env *value* (#11 `IMcpServer.envKeys` — names only). A field that does not exist cannot leak.
- **Never follow symlinks** on install, move or remove — exactly as `cleanup.ts:84` already does
  on delete. An install that follows a symlink writes outside the intended directory, and
  `writesTo` becomes a lie.
- **Install uses the same rigour as `checkDeletable`** (`cleanup.ts:55-123`): path containment,
  the check at the instant of the act (not of the inventory), failure as a *result* not an
  exception. It is unacceptable for the product to be more careful deleting a `node_modules`
  than writing a hook.

## 5. What an extension declares — its permission / capability surface (reused from #11, not redefined)

The install review can only show what the model can express. #11 already ratified the model;
this RFC **reuses it verbatim** and adds only the *derivation* an installer performs before
writing a byte. Nothing here redefines a Capability, an Extension or `requires`.

From the ratified taxonomy (#10) and model (#11), an Extension's surface is:

- **The Capabilities it provides** — `IExtensionManifest.provides`, a set drawn from
  `instruction | skill | command | subagent | prompt | hook | mcp-server` (#11 §5.1). A
  "plugin" is an Extension providing more than one Capability — *container by cardinality*
  (taxonomy §6.1). Cardinality and kind are what the review counts and ranks by severity.
- **The MCP servers it requires** — `IExtensionManifest.requiresMcp` (#11 §5.5): MCP-server
  Capabilities the Extension depends on without containing (taxonomy §5.3). *Nobody installs a
  requirement*, but the review must show it, because a required MCP server is filesystem and
  network reach the Extension assumes.
- **Each MCP server's reach** — `IMcpServer` (#11 §5.5): `transport`, `command`, `args`, and
  `declaredIn`. An MCP server is a process; its `command`/`args` are the shell it will run and
  the socket it may open. This is the highest-reach surface an Extension can carry.
- **The env variable *names* an MCP server reads** — `IMcpServer.envKeys` (#11 §5.5): **names
  only**, never values (§8). The review shows *which credentials a server will read from the
  environment*, classified by presence via `classifyEnvVar` (#45) as *configured / missing /
  inherited / externally stored* — never by value.

What the review **derives** on top of the manifest, and marks as fact or heuristic:

| Surface | What it is | Fact or heuristic |
|---|---|---|
| `writesTo` | the exact paths the install will create or touch | **Derived fact** — Blackfin is the writer, so this is the one claim it is certain of |
| `hooks` | shell scripts (`Capability(hook)`) that run without passing through a model | **Fact** (the kind is a disk predicate) — **high severity** |
| `mcpServers` | each required/provided `IMcpServer`, with `transport`/`command`/`args`/`envKeys` | **Fact** — high severity |
| `networkHints` | hosts/URLs found in the content | **Heuristic** — false positives expected (D3), marked as such |
| `shellHints` | command blocks inside skills/commands | **Heuristic** (D3), marked as such |

**Why axes are named after what Blackfin can measure, not after abstract reach (D2).** A trust
model that lists `filesystem / network / shell / secrets` as axes *sounds* rigorous, but
Blackfin cannot measure most of them: it cannot know what a persuasive skill will make the
agent read. Naming an axis you cannot measure is theatre. So the axes are `writesTo / hooks /
mcpServers / hints` — the things Blackfin *can* derive — each stamped **fact** or **heuristic**.
`writesTo` is the only axis Blackfin asserts with total certainty, precisely because it is the
one thing it does itself.

## 6. Provenance — facts, claims, and honest ignorance

Provenance is what Blackfin records about *where an item came from*. Each fact carries its own
certainty, and the absence of provenance is itself a first-class fact.

- **`source`** — from #11's `ExtensionSource`: `detected | installed-by-blackfin | marketplace
  | git | url`. Recall (#11 §5.3) that `detected` is **not a stored value** — it is the *absence*
  of an `IExtensionRecord`. Blackfin never invents provenance for a file it merely found.
- **`origin` / `sourceRef`** — the URL, git remote, and **commit/tag** the bytes came from. A
  **fact** when Blackfin did the download. An exact commit is the strongest provenance there is
  — and #13's federation keeps it that way, because the index *points at the origin* and never
  hosts the bytes (#13 D5).
- **`contentHashAtInstall`** — the hash at the instant of install (#11 §6). A **fact**, and the
  anchor for "you hand-edited this since you installed it" (`locallyModified`, #11 §8).
- **`installedAt` / `installedBy`** — **facts**.
- **Author / identity — NOT a fact.** A frontmatter `author` (or a catalog `claimedAuthor`,
  #13 §6.1) is a string anyone can type. The UI shows it as **claimed**, never as verified.
  #13 already carried this into the field name (`claimedAuthor`, not `author`) precisely because
  the marketplace is where a stranger's assertion is most likely mistaken for something Blackfin
  checked.
- **The absence of provenance is a first-class fact.** A `detected` item on disk has no origin,
  and Blackfin says *"this was detected on your disk — Blackfin did not install it and does not
  know where it came from."* It never guesses. Guessing provenance fabricates the ground the
  user decides on, and fabricated provenance is worse than none (taxonomy §5.4, #11 §5.3).

**What integrity verification would add, and where it is defined.** Provenance says *where from*;
integrity says *is it the same bit, signed by whom*. Those mechanics are **#51**, not this RFC.
This document only fixes what such verification would *prove*, so #51 knows what to build:

- **Checksum** against the registry-published value: *proves the bit did not change in transit —
  it does **not** prove the content is safe.* Both halves go in the same UI sentence, always.
- **Signature** (#51): proves key X signed. Worth exactly as much as trust in X — and since #13
  ratified that Blackfin operates no key, a signature only ever attests a *third party's* key.
- **TLS + host pinning**: proves where it came from.
- **Deterministic reinstall**: the same version yields the same hash. The cheapest, most
  underrated check.

The *how* of all four is #51. This RFC defines only what each would assert.

## 7. The install permission review (#50) — explicit, per-install, never generalized

The install review is the product's entire security surface (§2). This RFC defines its
*shape* — what the user is shown and must approve; the screens are #50, the flow through the
marketplace is #13 §7, the item page is #49.

### 7.1 What the review shows

Before writing a single byte, Blackfin reads the fetched artifact and shows, all of it
verifiable:

- **The capabilities added.** *"This extension declares 3 MCP servers, 2 hooks and 1 skill.
  Hooks run commands in your shell."* — counted from `provides` (#11), severity per §5.
- **The MCP servers and network reach.** Each `IMcpServer`: `transport`, `command`, `args`, and
  the env-var **names** it reads (`envKeys`, classified by presence, never by value). Plus the
  `requiresMcp` servers the extension assumes exist. Plus `networkHints`/`shellHints`, **marked
  as heuristics** (D3).
- **The exact paths it will write.** *"It will write to `~/.claude/skills/foo/` and nothing
  else."* — from the derived `writesTo` (§5, §6). The one claim Blackfin is certain of.
- **The source and provenance.** *"Came from `github.com/foo/bar`, tag `v1.2.0`, commit
  `abc123`. Blackfin did **not** verify who `foo` is."* Or, for a detected item: *"detected on
  your disk — Blackfin did not install it and does not know where it came from."*
- **What was and was not integrity-checked.** *"The checksum matches what the registry
  published. That proves the file did not change in transit — it does not prove the file is
  safe."* (Mechanics: #51.)
- **The irreversible / hard-to-reverse parts, named as such.** A hook that runs on the next
  agent session; an MCP server that, once configured, the agent may connect to on its own; a
  write into a shared global path that affects every project. The review says which parts are
  hard to undo *before* the yes, not after.
- **What Blackfin cannot do, in plain words** (§9): *"After this is installed, the agent runs
  it, not Blackfin. Blackfin cannot restrict what it does."*

And — layer (D), §4.4 — **the agent's own permission model where it exists**, and the honest
negative where it does not: *"Claude Code will gate this hook behind its `allowed-tools`
prompt"* versus *"this agent has no permission model; nothing gates what this does."*

### 7.2 Proportional friction, not uniform fear

Friction is proportional, not uniform. Installing a plain-text skill that only adds context
**must not** carry the same dialog as installing an extension with two shell hooks. Two levels
(D4): a **simple review** (plain text, `severity: 'low'`) and a **capability review** (hooks,
MCP, shell, `severity: 'elevated'`). If every dialog is equally frightening, the user learns to
ignore all of them — and the product's only defence cancels itself. A heuristic false positive
(a URL in a code comment shown as network reach) trains exactly that reflex, which is why D3 —
whether to show heuristics at all — is a real fork, not a detail.

### 7.3 The decision — explicit, recorded, revocable, and re-asked when reach widens

- **The user approves an extension at a version, with a list of capabilities.** Consent is
  **per install**, never generalized — no "install everything," no "trust this author forever."
- **An update that *adds* a capability requires fresh approval.** The v1.3 that introduces a
  hook is a new trust decision, not a patch. Silent capability-widening on update is the classic
  attack route (#13 §8). The mechanism is `approvedCapabilitiesHash` (D6): a hash of the approved
  capability *set*, so a v1.3 that only fixes a typo does not re-ask, but one that adds a hook
  does.
- **The decision lives in `IExtensionRecord.trust`** (#11 reserved the slot as `unknown`; §10
  fills it), in **Blackfin's own Dexie database** — never in the user's files, never in the
  throwaway inventory cache that `pruneTo` deletes (#11 §7, `workspace-database.ts:73-85`).
- **Revoking = uninstalling through the safe flow of #30**: revalidate, confirm the path,
  identify dependencies, report impact, trash, record the outcome — the rigour of `checkDeletable`
  (`cleanup.ts:55-123`).

## 8. Secrets — a type invariant, not a discipline

MCP `env` blocks routinely hold tokens. The rule is absolute and it is enforced by the **type**,
not by care:

- **No env-variable *value* is ever read, shown, logged or persisted.** The MCP-server type
  (`IMcpServer`, #11 §5.5) has **no field** capable of holding a value — only `envKeys`, the
  **names**. A field that does not exist cannot be filled by mistake, logged by mistake, or
  persisted by mistake. That absence *is* the control.
- **A secret is reported only by presence**, via `classifyEnvVar` (`app/src/lib/workspace/mcp/secrets.ts`,
  #45/#86): *configured*, *missing*, *inherited*, or *externally stored*. The function computes
  presence from the *shape* the config declared and an injected `environmentHasName` that checks a
  key's existence (`!== undefined`) without reading it. `process.env` is never copied anywhere.
- **URLs are sanitized before display** (`sanitizeMcpUrl`, same file): userinfo, query string,
  fragment and credential-shaped path segments are stripped; only `scheme://host[:port]/path`
  survives, and even an unparseable URL yields no excerpt (the raw string may *be* the credential).
- **No token, credential or secret is persisted in any Blackfin database.**
  `IExtensionRecord.trust` (§10) has no slot for one, and review of this RFC must reject any field
  that opens one.

## 9. What Blackfin cannot do (said here, and said in the UI)

This is the most important deliverable of the document. It must appear in the RFC *and* in the
install screen, in plain language:

- **It cannot stop an installed skill from instructing the agent to read `~/.aws/credentials`.**
- **It cannot stop a hook from running any command in the user's shell.**
- **It cannot stop an MCP server from opening any socket.**
- **It cannot verify the declared author is who they claim to be.**
- **It cannot detect malicious code. It is not an antivirus.**
- **It cannot contain, sandbox, or restrict anything after install.** The agent runs it.

And, so the honesty is not only negative — **what Blackfin can do, and no other tool does:**

- Show **every** context item affecting a project, including the global ones invisible from
  inside it (the founding argument: `workspace-inventory.ts:63-70`).
- Enumerate exactly what an install will write, **before** it writes (`writesTo`).
- Distinguish what it installed from what it merely found (`source`), and admit when it does not
  know (`unknown-detected`).
- Detect post-install hand-editing (`contentHashAtInstall`, #11).
- Require fresh approval when an update adds a capability (`approvedCapabilitiesHash`).
- Surface the agent's own permission model where it exists — and say so when it does not (§4.4).

## 10. Data model (normative appendix — implemented by #50/#35, not created here)

No data is persisted by this issue (it is a document). What it defines is the content of
`IExtensionRecord.trust`, left `unknown` by #11, and the disclosure computed at install. The
sketch below reuses #11's `IMcpServer` and `ExtensionSource`; it does not redefine them.

```ts
// Normative appendix. Implemented by #50 and #35. Reuses #11's IMcpServer,
// ExtensionSource, contentHashAtInstall — nothing here redefines the domain model.

/**
 * What Blackfin can DERIVE from an artifact before writing it to disk.
 * Note what is NOT here: no promise of containment, no boolean field called
 * `safe`. Each item is a fact or an explicitly-marked heuristic.
 */
export interface IExtensionDisclosure {
  /**
   * The exact paths the install will create or touch.
   * DERIVED BY THE INSTALLER, never declared by the author: the one claim in
   * this list Blackfin is certain of, because it is the one doing the writing.
   */
  readonly writesTo: ReadonlyArray<string>
  /** Shell scripts. Run without passing through a model. High severity. */
  readonly hooks: ReadonlyArray<{ readonly path: string; readonly command: string }>
  /** Declared/required MCP servers. `envKeys` are NAMES; values are never read (#11). */
  readonly mcpServers: ReadonlyArray<IMcpServer>
  /** HEURISTIC. Hosts found in the content. False positives expected (D3). */
  readonly networkHints: ReadonlyArray<string>
  /** HEURISTIC. Command blocks inside skills/commands (D3). */
  readonly shellHints: ReadonlyArray<string>
  readonly severity: 'low' | 'elevated'
}

/** Facts about origin. Each carries its own certainty. */
export type IProvenance =
  | {
      readonly kind: 'blackfin-installed'
      readonly source: ExtensionSource //   #11: marketplace | git | url | installed-by-blackfin
      readonly origin: string //            fact: we downloaded it
      readonly ref: string | null //        commit/tag: fact
      readonly contentHashAtInstall: string // fact (#11)
      readonly checksumVerified: boolean //  fact: matches the registry (mechanics: #51)
      readonly signatureVerified: boolean // fact: signed by key X (mechanics: #51)
      /** CLAIMED by the manifest/catalog. Never verified. The UI must say so. */
      readonly claimedAuthor: string | null
    }
  /**
   * It was on disk. Blackfin DOES NOT KNOW where it came from, and says so.
   * Guessing provenance fabricates the ground the user decides on.
   */
  | { readonly kind: 'unknown-detected' }

/** The user's decision. Recorded in BLACKFIN's database, never in the user's files. */
export interface ITrustDecision {
  readonly decidedAt: number
  readonly version: string | null
  /**
   * Hash of the APPROVED CAPABILITY SET (see D6). If an update changes this hash
   * — a new hook — Blackfin must ask again. Silent capability-widening on update
   * is the classic attack route.
   */
  readonly approvedCapabilitiesHash: string
  readonly provenance: IProvenance
  // DOES NOT EXIST: `sandboxed`, `safe`, `verified` — no direct object.
  // DOES NOT EXIST, EVER: token, credential, secret, env value.
}
```

Where it lives: `IExtensionRecord.trust` (#11 §6), in the **independent Blackfin Dexie
database** — never in the inventory cache (which `pruneTo` deletes,
`workspace-database.ts:73-85`), and never in a user file.

## 11. Security

The whole document is the security section. Stated unambiguously:

- **Blackfin never executes an extension, and never connects to an MCP server, as a side
  effect of a trust decision.** Reviewing, disclosing, recording provenance, approving — none of
  it runs anything or opens a socket. Execution is the agent's job. Testing a connection to an
  MCP server *is* executing, and is explicitly out of scope (it is #46, opt-in and per-item).
- **Trust is never inferred silently.** `source` is never guessed (`detected` = absence of a
  record, #11 §5.3); a decision is recorded only after an explicit yes; a detected item Blackfin
  did not install is reported as such, never quietly "trusted."
- **No secret is ever read, shown, or persisted** (§8). The guarantee is of **type**: no field
  can hold an env value.
- **Never follow symlinks** on install, move or remove — the rule `cleanup.ts:84` already applies
  on delete.
- **Never delete irreversibly by default** — install and removal go through #30's safe flow.
- **The CLI is a trust surface (D8).** When an agent can call `blackfin` (M5, #65), it becomes a
  write path *into* the app. An agent that can install extensions can close its own compromise
  loop. This RFC nails the boundary **before** #65 is written: **an agent may never install an
  extension without a human confirmation in the UI.** No CLI path may auto-approve a trust
  decision.
- **A federated-marketplace item is no safer than any Git repo** (#13 §10). Because #13 ratified
  that Blackfin curates and signs nothing, the review must make "easier to find ≠ vouched for"
  uncomfortably clear. No badge may suggest otherwise.

## 13. Out of scope

- **Promising, specifying or implementing an OS-level sandbox.** Explicitly forbidden by the
  product boundary and demonstrated non-viable (§4.1). A proponent must open a separate issue
  with a running proof of concept, not an RFC section.
- **Executing, hosting or supervising agents, MCP servers or extensions.** Blackfin is not an
  agent host.
- **Testing a connection to an MCP server.** Connecting is executing; it is #46, opt-in and
  per-item, never automatic.
- **Turning on `contextIsolation` / turning off `nodeIntegration`** (`app-window.ts:78-80`). A
  whole-fork refactor, and a prerequisite of nothing here.
- **Antivirus, static analysis of extension code, malware detection.** Blackfin is not a threat
  scanner and must not pretend to be.
- **The integrity/signature key chain** — #51. **The registry** — #13. **Org policy/allowlists**
  — #53. **UI screens** — #50/#49. Referenced, never redefined.

## 14. DECISÕES RATIFICADAS

**RATIFICADO pelo mantenedor.** **D1 = disclosure sobre contenção** (§4.4): a Blackfin mostra o
que uma extensão adiciona, de onde veio e o que ela não sabe, e exige revisão explícita por
instalação — mas **nunca afirma que é segura**. Sandbox próprio e confiança binária rejeitados;
contenção real fica com o modelo de permissão do agente; reputação/allowlist é política (#53),
nunca afirmação de segurança. As demais decisões (D2–D8) seguem de D1 e ficam adotadas com as
recomendações abaixo. #50/#51/#53/#38/#65 implementam sobre isto. A recomendação original casou

explícita*?** A recomendação é a segunda — a Blackfin mostra o que a coisa é e de onde vem, e
nunca diz que é segura.
