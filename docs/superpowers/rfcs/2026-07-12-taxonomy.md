# RFC — What a Plugin, a Skill, and an MCP server actually are

- **Date:** 2026-07-12
- **Issue:** [#10](https://github.com/matbrgz/blackfin/issues/10) — *RFC: Define what a Plugin, a Skill, and an MCP server actually are*
- **Status:** Proposal. Adopts **Option C** (Extension + Capability). Awaiting maintainer ratification of **D1** (see "DECISÃO PENDENTE").
- **Blocks:** #11 (types), and transitively #12 (trust), #13 (registry), #14 (persistence), #21 (migration), #43 (MCP content). Feeds #16 (UI section naming) without blocking it.
- **Scope of this document:** vocabulary only. No code. No `ContextRole` change, no `catalog.ts` change, no inventory change.

---

## 1. Problem

Blackfin already reads, classifies and displays the configuration of fourteen coding
agents — but it has no vocabulary for what it is reading. Today everything an agent
reads collapses into one flat enum, `ContextRole` (`app/src/models/workspace-inventory.ts:32-47`):
seven members — `Instructions`, `Skill`, `Command`, `Subagent`, `Prompt`, `Settings`,
`Hook`. There is no **Plugin**. There is no **MCP server**: the only trace of MCP in
production code is the string `'mcp.json'` inside `SettingsFiles`
(`app/src/lib/workspace/catalog.ts:182`), which makes a file that declares five MCP
servers — with commands, arguments, environment variables and network reach — classify
as "Settings" and never have its content read. There is even a generated `mcp` icon
(`app/src/ui/octicons/octicons.generated.ts:3460`) that nothing uses.

This vocabulary is the root of the whole backlog. #11, #12, #13, #43 and the marketplace
all describe operations on nouns — "install a plugin", "disable a skill", "review an
MCP's permissions" — that **are not yet defined**. Writing any of them first is writing
fiction.

And the problem is not internal. The agents Blackfin observes **disagree with each
other** about what these words mean:

- **Claude Code** has plugins that are containers of skills, commands, subagents, hooks
  and MCP servers — a plugin marketplace, with a `plugin.json`.
- **Cursor** has *rules* (`.cursor/rules/*.mdc`) and MCP, and no notion of plugin or skill.
- **Copilot** has *instructions* and *prompts* under `.github/`, and no home directory of its own.
- **Codex / OpenCode / Amp** share the `AGENTS.md` convention, which belongs to no agent.
- **Continue** calls *blocks* (models, rules, MCP servers) what Claude would call a plugin.
- **Goose** calls *extensions* exactly what the others call an MCP server.
- The `npx skills add` convention installs into `.agents/skills/`, with no owner.

No single term is stable across them. Yet Blackfin needs **one** model, because the
screen is one.

## 2. Why the vocabulary matters

A wrong taxonomy is not a naming detail — it is a cost paid forever:

- If **Plugin** is modelled as a peer of Skill, a Claude Code plugin containing three
  skills shows up as four loose items, and uninstalling it orphans the rest.
- If **Skill** is modelled as agent-specific, the same `SKILL.md` copied into
  `~/.claude/skills/` and into `.agents/skills/` shows up twice, unrelated, and the user
  cannot tell it is the same thing.
- If **MCP server** is modelled as an installable extension, the marketplace will offer
  to "install the GitHub MCP" when what the user actually needs is for *a plugin to
  declare that it depends on it* — and the permission list shown at install time becomes
  a list about the wrong object.

The product boundary states it directly: *a Plugin is not trusted code by default; an
MCP server is more than a name and a URL; a Skill is more than a Markdown file — it has
an origin and a scope.* None of those three sentences can even be expressed in the
current model.

## 5. The canonical noun set

Two nouns, one relationship. No third ecosystem noun is invented — Blackfin normalises
what exists, it does not propose a competing standard.

### 5.1 Extension — the unit of distribution, trust, versioning

- **What it is:** the thing you install, version, trust and remove; it has identity, an
  origin, a version and a manifest.
- **Disk predicate:** a manifest declaring it (e.g. Claude Code's `plugin.json`). A
  hand-written `.cursorrules` is **not** an Extension — it is detected configuration. No
  manifest ⇒ no Extension.
- **Unit of identity:** its manifest identity — `id` (name + origin), with `version` as
  a distinct axis. Two installs of "superpowers 1.2" and "superpowers 1.3" are the same
  Extension at two versions, not two Extensions.
- **Relations:** **contains** one or more Capabilities (`provides`); **requires** zero or
  more MCP-server Capabilities it does not contain (`requiresMcp`). It does not contain
  what it merely requires.

### 5.2 Capability — the unit of effect

- **What it is:** anything that changes the agent's behaviour —
  `instruction | skill | command | subagent | prompt | hook | mcp-server`.
- **Disk predicate:** per kind, the predicate already used by the classifier. Canonical
  example: **Skill = a directory containing `SKILL.md`** (`catalog.ts:204-206`) — keyed
  off the manifest, so a skill is a skill only once it declares itself. The predicate is
  the criterion of existence: Blackfin never invents an item the disk does not sustain.
- **Unit of identity:** **content-based**, not path-based — `contentHash + name`
  (**D4**). Two copies of the same `SKILL.md` in `~/.claude/skills/pdf/` and
  `.agents/skills/pdf/` are **the same Capability in two places**, differing only in
  `scope` and in which `agents` read it. This is precisely what the current model cannot
  say: `IContextFile` is identified by `relativePath` (`workspace-inventory.ts:99`).
- **Relations:** **belongs to** exactly one Extension when installed (`extensionId`), or
  to **none** when merely detected (`extensionId = null`); **is read by** one or more
  agents (`agents: AgentId[]`); an `mcp-server` Capability **is required by** zero or
  more Extensions.

Every Capability carries: `kind`, `agents: ReadonlyArray<AgentId>` (more than one ⇒
portable), `scope` (`Global | Project`, `workspace-inventory.ts:71-76`), `relativePath`
(still the truth of where it is), and `extensionId: string | null`.

### 5.3 `requires` — the deliberately-not-containment relationship

An Extension may **require** an MCP-server Capability without **containing** it. The MCP
server can exist independently — it is usually four lines in a JSON file that already
exists, not a downloaded artifact. Nobody "installs" a requirement. This is why MCP is a
Capability and never an Extension (§6.3).

### 5.4 The boundary line, materialised in the type

`extensionId: null` **is** the product rule "detected filesystem config and
Blackfin-managed data never mix", made concrete in the type. It is filled **only** when
Blackfin installed the item — never by inference. Guessing that a skill "probably came
from plugin X" fabricates provenance, and fabricated provenance is worse than none.

## 6. The three central decisions

### 6.1 Is a Plugin a container or a peer? → **Container, but not a kind.**

A Plugin is an `Extension` whose manifest declares **more than one** `Capability`. There
is no `kind: 'plugin'` sibling of `kind: 'skill'`; there is an Extension that *contains*
capabilities, one of which may be a skill. "Container by cardinality, not by special
type." This dissolves the question instead of answering it, which is the point.

- **Rejected — Plugin as a peer kind (Option B's flattening):** a Claude Code plugin
  with three skills would surface as four loose items, and uninstalling would orphan the
  rest. It also strips the plugin of the identity — name, version, changelog, single
  trust decision — that belongs to the container, not to each skill. The marketplace
  (#13) would be left with nothing to sell.

### 6.2 Is a Skill portable or agent-specific? → **The artifact is portable; the install is not.**

The `SKILL.md` (frontmatter + body) is identical in `~/.claude/skills/` and
`.agents/skills/`. What differs is *where it is* and *who reads it*. Therefore a
Capability carries `agents: ReadonlyArray<AgentId>` (who reads it) and `scope` (how far
it reaches), and logical identity is `contentHash + name`, **not** the path. Two copies
in different scopes are **one Capability in two places**, and the UI can say so.

- **Rejected — Skill as agent-specific (path identity):** the same file in two scopes
  would appear as two unrelated skills, and the user could never learn they are the same
  thing. This is exactly the failure the current `relativePath` identity
  (`workspace-inventory.ts:99`) produces, and #10 exists to end it. See **D4**.

### 6.3 Is an MCP server an Extension or a required Capability? → **It is a Capability, and it is the target of `requires`.**

An MCP server changes what the agent can do (so it is a Capability) and an Extension can
depend on it without containing it (so it is a `requires` target). It is honestly both,
and Option C is the only model that lets it be both without contradiction.

- **Rejected — MCP as an installable Extension:** installing an MCP server is almost
  always *writing four lines into a JSON that already exists*, not downloading an
  artifact. Modelling it as an Extension would make the marketplace offer "install the
  GitHub MCP" and would hang the install-time permission list on the wrong object. See
  **D5**.

Two normative consequences this RFC fixes now:

- **An MCP server is not a file — it is an entry inside one.** `catalog.ts:182`
  classifies the container (`mcp.json` as `Settings`); the taxonomy obliges modelling the
  content. That reshaping is the expensive part and is all of #43.
- **Env values are never read.** An MCP `env` routinely holds tokens. As a domain
  invariant — not an implementation detail — an environment-variable **value** is never
  read, shown, logged or persisted; a secret is reported only as *configured*, *absent*,
  *inherited* or *externally stored* (#45). The data model must not even have a field
  able to hold the value.

## 7. The five user-facing sentences (the model as a test)

The RFC fixes the sentences the screens must be able to say. If the model cannot say
them without ambiguity, the model is wrong. Each resolves cleanly under Option C:

1. *"This skill came from the **superpowers** plugin, installed from a marketplace, in
   **global** scope. Removing it also removes 3 commands and 1 hook."*
   → Capability`(skill)` with `extensionId = superpowers`; the Extension `provides`
   `[skill, command×3, hook]`; `scope = Global`. Removal is an operation on the
   Extension, and the count is `provides`. **Sayable.**

2. *"The MCP server **github** is declared in two places: `~/.claude/settings.json` and
   `.cursor/mcp.json`. Are they the same? Blackfin says **yes/no** — and says why."*
   → Two Capability`(mcp-server)` occurrences; identity by `contentHash + name` over the
   server entry (name + command + args). Same ⇒ one Capability read by
   `agents = [ClaudeCode, Cursor]` in two places; the "why" is the content match.
   **Sayable.**

3. *"This `SKILL.md` exists in `~/.claude/skills/pdf/` and in `.agents/skills/pdf/`. Is
   it the same skill in two scopes, or two different skills?"*
   → One Capability`(skill)` (`contentHash + name` match), two `scope`s. **Sayable** —
   and unsayable under path identity, which is why **D4** matters.

4. *"This plugin depends on the MCP server **filesystem**, which is not configured. It
   will not work."*
   → Extension with `requiresMcp = ['filesystem']`; no Capability`(mcp-server)` named
   `filesystem` is present ⇒ unmet requirement. **Sayable** — and impossible if MCP were
   an Extension (§6.3).

5. *"Nothing in this section is run by Blackfin. The agent runs it."*
   → A product invariant (§9), not a model feature. **Stated, textually.**

## 8. What the taxonomy refuses to model, and why

Blackfin observes and organises; it is not an agent host. The taxonomy deliberately does
**not** model:

- **Trust, permissions, provenance (#12).** The taxonomy makes trust *expressible* —
  `Extension` is the subject a trust decision hangs on — but assigns none. Recognising a
  plugin says nothing about whether it is trusted.
- **Registry, distribution, versioning (#13).** The marketplace, its catalog and update
  flows are out. `version` exists as a field so #13 has somewhere to write; the registry
  itself is #13's.
- **Persistence (#14).** This RFC persists nothing but its own document. No install
  records, no database shape.
- **Execution / hosting.** Blackfin never runs an Extension, connects to an MCP server,
  fires a hook, or tests a plugin. The agent runs it. This is a permanent product
  property, not a temporary limitation (§9).
- **Rewriting the user's files.** Defining a taxonomy does not authorise normalising a
  user's files to fit it. If a `.cursorrules` does not fit, it is reported as-is — the
  model bends to the disk, never the reverse.
- **Silent provenance inference.** `extensionId` is filled only when Blackfin installed
  the item; it is never guessed (§5.4).
- **Inventing new ecosystem nouns.** Only `Extension` and `Capability`. No competing
  standard.
- **UI section naming.** Whether the user sees the word "Extension" or "Plugin" is #16
  and **D2**.

## 9. Product invariants this RFC states textually

- **Blackfin does not execute extensions.** Recognising an MCP server, a hook or a
  plugin never implies executing it, connecting to it, or testing it. Whoever executes is
  the agent.
- **The disk predicate is the criterion of existence.** A skill is a skill because it has
  `SKILL.md` (`catalog.ts:204`), not because a database says so.
- **`AgentId.Shared` remains.** The current model got this right
  (`workspace-inventory.ts:9-13`); nothing here revokes it.
- **Detected config and managed data never mix.** Materialised as `extensionId: null`.
- **Env values never enter the model.** No field can hold an environment-variable value.

## 10. Options considered

Summarised from issue #10; the full trade-offs live there.

- **Option A — Mirror Claude Code.** Adopt Claude Code's taxonomy as canonical (Plugin
  is the container; Skill is child-or-loose; MCP lives in a plugin or in settings).
  *For:* 1:1 with the richest ecosystem, zero translation for the most-used agent.
  *Against:* makes the other thirteen agents second-class *by construction*; Blackfin's
  domain moves whenever Claude Code's does. *Cost of error:* high and permanent.
- **Option B — Flat capabilities, no container.** No Plugin in the domain; only
  capabilities, with `origin` as a field. *For:* survives all fourteen agents, keeps
  `ContextRole` nearly as-is, cheap #21. *Against:* loses plugin *identity* (name,
  version, single trust decision); "update plugin X 1.2→1.3" has no subject; the
  marketplace has no object to sell. *Cost of error:* medium.
- **Option C — Two explicit layers: Extension + Capability.** *(Recommended.)* Each of
  the three central questions gets an answer that does not force the others. *Cost of
  error:* low — if `Extension` proves useless it degrades to an `origin` field and
  becomes Option B without data loss.
- **Option D — Do not decide; model per agent, unify later.** *For:* zero risk of
  choosing wrong today. *Against:* the ecosystem will not converge, and "unify later"
  means unifying over already-persisted install data; the cross-project screens need a
  common domain *now*. *Cost of error:* guarantees rework.

### Recommendation

**Option C.** The three central questions have answers that differ from one another, and
only C lets each be given without one contaminating another (§6). It respects the product
boundary literally: `extensionId: null` *is* the line between detected config and managed
data. It makes trust expressible (the security reason to prefer C over B): without
`Extension` as a subject, #12 has nothing to hang provenance and permissions on.

## 12. Fixtures corpus

The classification contract lives beside this document, under
`docs/superpowers/rfcs/fixtures/taxonomy/` — one anonymised directory tree per agent, as
a `<agent>.tree.txt` showing the real directory structure and representative file names.
Every tree is small, clearly synthetic, and contains **no secret values** (MCP `env`
keys appear with placeholder markers, never real tokens). These are the cases any
implementation of the taxonomy must classify correctly. The corpus deliberately includes
the cases that break a naive model:

- A Claude Code plugin with skills, commands and an MCP server declared inside it → **1
  Extension, N Capabilities**, not N+1 loose items.
- The **same** `SKILL.md` in `~/.claude/skills/pdf/` and `.agents/skills/pdf/` → the
  **D4** test: one Capability or two.
- An `mcp.json` declaring three servers with different transports (stdio, http) → today a
  single `Settings` blob (`catalog.ts:182`); the taxonomy demands three
  Capability`(mcp-server)`.
- The same MCP server (same name, same command) in `~/.claude/settings.json` **and**
  `.cursor/mcp.json` → the cross-agent identity test (sentence #2).
- An `AGENTS.md` at root, belonging to no agent (`AgentId.Shared`).
- A hand-written `.cursorrules` — no plugin, no manifest, no version → the orphan
  (`extensionId: null`) that is most of the real world.
- A skill with malformed frontmatter → `parse.ts:29-69` already returns
  `{ name: null, description: null }` rather than throwing; the taxonomy must not demand
  more.

See `docs/superpowers/rfcs/fixtures/taxonomy/README.md` for the per-agent index and the
expected classification of each tree.

## Appendix — normative type sketch (NOT to be merged by this issue)

For #11 to formalise and #21 to implement. Included to make the recommendation
verifiable rather than rhetorical.

```ts
/** What changes an agent's behaviour. The unit of effect. */
export enum CapabilityKind {
  Instruction = 'instruction',   // CLAUDE.md, .cursorrules, AGENTS.md
  Skill = 'skill',               // a directory with SKILL.md
  Command = 'command',
  Subagent = 'subagent',
  Prompt = 'prompt',
  Hook = 'hook',
  McpServer = 'mcp-server',      // NEW: absent from the domain today
}

export interface ICapability {
  readonly kind: CapabilityKind
  /** Which agents read this. More than one ⇒ portable. */
  readonly agents: ReadonlyArray<AgentId>
  readonly scope: ContextScope
  /** Where it is on disk. Still the truth. */
  readonly relativePath: string
  /**
   * The Extension that installed this — or null, if only DETECTED on disk.
   * This line is the boundary between user config and Blackfin data.
   * It is never filled by inference.
   */
  readonly extensionId: string | null
}

/** What you install, version, trust and remove. The unit of distribution. */
export interface IExtension {
  readonly id: string
  readonly name: string
  readonly version: string | null
  /** Capabilities THIS extension provides. A "plugin" is cardinality > 1. */
  readonly provides: ReadonlyArray<CapabilityKind>
  /** MCP servers it DEPENDS on without containing. Nobody installs a requirement. */
  readonly requiresMcp: ReadonlyArray<string>
}
```

Logical identity of a Capability is `contentHash + name` (**D4**), not `relativePath`.
The exact `IMcpServer` shape (transport, `command`, `args`, `env`) is #11/#43 — with the
invariant already firm here that `env` **values** are never read, shown or persisted;
only the presence of a key is.
